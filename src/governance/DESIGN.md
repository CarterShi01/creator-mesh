# Design: src/governance

## Current Design Summary

`governance` is the control, safety, and audit layer. It enforces human-in-the-loop by requiring creator approval before significant actions proceed. It also maintains an append-only audit trail. No implementation files exist yet. The key invariant: actions classified as medium risk or above must not proceed without an `ApprovalResult`.

## Design Goals

- Ensure the creator can review, approve, reject, or defer any significant action before it executes.
- Maintain a tamper-evident audit trail of all significant decisions and state changes.
- Support configurable approval policies without requiring code changes.

## Key Decisions

- **Approval is mandatory for medium risk and above**: governance is not optional middleware. Any action at medium risk or higher must pass through an `ApprovalCheckpoint`.
- **Audit records are append-only**: past records must not be modified. This is a core safety invariant.
- **Policies are externally configurable**: `ApprovalPolicy` and `PermissionPolicy` should be data-driven, not hardcoded, so the creator can adjust thresholds without deploying code.
- **`governance` does not call back into orchestrator**: it receives approval requests and returns results. Calling back would create a circular dependency.

## Tradeoffs

- Requiring approval for every medium-risk action creates friction. Too broad a definition of "medium risk" will make the system feel like it asks for permission constantly. The `RiskLevel` taxonomy must be carefully calibrated.
- Append-only audit records are safe but require a compaction or archival strategy over time to prevent unbounded storage growth.

## Alternatives Considered

- **Inline approval in each layer** — rejected. Distributing approval checks across orchestrator, workflows, and runners makes the approval model inconsistent and hard to audit.
- **Optimistic execution with post-hoc audit** — rejected. CreatorMesh deals with personal notes, external messages, and code. Executing first and auditing later is not acceptable for significant actions.

## Current Assumptions

- At v1, approval will be presented as a prompt to the creator in the same terminal or interface where they issued the command.
- `RiskLevel` will start with three levels: `low` (no approval needed), `medium` (approval required), `high` (approval + explicit confirmation required).
- Cost limits (`CostLimit`) will cap token usage per session. The first limit will be a configurable daily token budget.

## Connector Action Audit Integration (resolved from src/connectors design)

Connector actions are now a defined category of auditable side effects. Every call to `ConnectorPort.execute()` produces a `ConnectorAction` record that flows into governance as an `AuditRecord`.

Key implications for governance design:
- `ApprovalRequirement` (`never | conditional | always`) is declared per capability in `CapabilityRegistry` and checked by the orchestrator before calling `execute()`. Governance receives the pre-decided `approvalResult` as part of `ConnectorAction`.
- The `auditId` in `ConnectorResult` is a UUID generated by governance when the `AuditRecord` is persisted.
- Connector actions must appear in the audit trail regardless of status — including auto-approved, rejected, and failed actions.
- Permission levels (`safe-read | write | destructive | external-side-effect`) from the connector capability model map onto governance risk levels. `destructive` and `external-side-effect` require explicit approval.

## Open Questions

- How is an `ApprovalRequest` presented to the creator — CLI prompt, a notification, a UI modal? The presentation layer depends on the eventual interface, which is not yet designed.
- Should governance log all actions (including low-risk) or only medium-risk and above? Full logging is safer but noisier.
- How does the system handle an expired or unanswered approval request — auto-reject after timeout, or hold indefinitely?

## Future Evolution

- A governance dashboard may emerge to let the creator review pending approvals and audit history.
- Policy configuration may evolve from a simple file to a structured rule engine as CreatorMesh becomes more powerful.
- Multi-user governance (approvals from multiple stakeholders) may be needed for team contexts.
