# Interface: src/triggers

## Purpose

`src/triggers` is the interaction boundary of CreatorMesh. It receives internally generated thoughts, externally triggered messages, user actions, external events, and system signals, then represents them as stable input primitives for downstream creation, knowledge, workflow, and execution layers. It does not interpret user intent, construct creator objects, execute workflows, or call external systems.

## Public Concepts

### Input Primitives (Implemented)

**`Thought`** — an internally generated idea, note, reflection, insight, plan, or creative spark.

```ts
type ThoughtSource = "manual";

interface Thought {
  id: string;        // UUID, generated at creation
  content: string;   // trimmed; non-empty enforced
  createdAt: Date;   // defaults to current time
  source: ThoughtSource;
}

function createThought(content: string, source?: ThoughtSource): Thought
```

Invariants:
- `content` is always trimmed.
- Empty `content` (after trim) throws `Error("Thought content must not be empty.")`.
- `id` is a UUID generated by `crypto.randomUUID()`.
- `createdAt` defaults to `new Date()` at call time.
- `source` defaults to `"manual"`.

**`Message`** — an externally triggered input such as a request, feedback, opportunity, or task.

```ts
type MessageSource = "manual";

interface Message {
  id: string;        // UUID, generated at creation
  content: string;   // trimmed; non-empty enforced
  createdAt: Date;   // defaults to current time
  source: MessageSource;
}

function createMessage(content: string, source?: MessageSource): Message
```

Invariants:
- `content` is always trimmed.
- Empty `content` (after trim) throws `Error("Message content must not be empty.")`.
- `id` is a UUID generated by `crypto.randomUUID()`.
- `createdAt` defaults to `new Date()` at call time.
- `source` defaults to `"manual"`.

### Signal Types (Planned — not yet implemented)

- `ThoughtTrigger` — a creator-initiated signal carrying a thought, idea, note, or reflection
- `MessageTrigger` — an externally received signal carrying a message, request, or opportunity
- `ScheduledTrigger` — a time-based signal such as a daily review or weekly planning prompt
- `SystemEventTrigger` — a signal raised by a tool or workflow state change
- `TriggerPayload` — a common envelope wrapping any trigger type for downstream processing

### Normalization Types (Planned — not yet implemented)

- `NormalizedInput` — a cleaned and structured representation of a raw trigger payload
- `SourceMetadata` — metadata about the origin of an input (source type, timestamp, language, channel)
- `CapturedInput` — the fully normalized output ready for downstream processing

## Inputs

Raw signals from the outside world or from the creator:
- Manual thought or note entry
- Incoming messages from external sources
- Scheduled jobs
- System or tool state events

## Outputs

Stable input primitives (`Thought`, `Message`) and trigger envelopes for downstream layers (`creation`, `knowledge`, `orchestrator`, `workflows`).

## Allowed Dependencies

- `src/shared` (if strictly necessary)

## Disallowed Dependencies

`src/triggers` must not import from any higher-level module:

- `src/creation`
- `src/knowledge`
- `src/orchestrator`
- `src/agents`
- `src/runners`
- `src/connectors`
- `src/workflows`
- `src/governance`
- `src/storage`
- `src/outputs`

External libraries should only be used if strictly required for input type definitions, and only with approval.

## Invariants

- `src/triggers` must not import from higher-level modules.
- `src/triggers` must not contain business logic, agent prompts, or tool-specific code.
- Types defined in `src/triggers` should change slowly and deliberately. Breaking changes require cross-layer review.
- `createThought()` and `createMessage()` must always trim content and reject empty strings.

## Main Files

- `thought.ts` — `Thought` interface, `ThoughtSource` type, `createThought()` function
- `message.ts` — `Message` interface, `MessageSource` type, `createMessage()` function
- `index.ts` — barrel re-export of all public types and functions

## Change Rules for Agents

1. Read `AGENTS.md`.
2. Read `docs/context-map.md`.
3. Read this `INTERFACE.md`.
4. Read this directory's `README.md`.
5. Identify whether the change affects public concepts, inputs, outputs, dependencies, or invariants.
6. Update this file if the public contract changes.
7. After any change to Thought or Message types, check downstream modules that import from `src/triggers`.
