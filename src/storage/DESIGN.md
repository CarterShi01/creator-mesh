# Storage Module Design

Phase 1.5 implementation of the persistence layer described in `docs/architecture.md`. Uses SQLite (better-sqlite3) as the first real backend.

## Convergence map

| Phase 1 concept | Storage type | Phase 0 target |

## Architecture boundaries (portability constraints)

These invariants ensure Phase 2 can swap SQLite for Postgres, Mongo, or a graph DB by adding a new adapter directory:

1. **All SQL confined to `src/storage/sqlite/`** — no SQL strings outside that directory. The architecture-boundaries harness enforces this.
2. **Ports expose domain types only** — `WorkflowRunRecord`, `WorkflowDefinitionRecord`, `ManagedProject`, `RelationRow`. No cursors, prepared statements, or row objects cross the port boundary.
3. **Complex aggregations are port methods** — `planProgress()` is on `WorkflowRunStore`; the SQLite adapter implements it via a view. A future Mongo adapter would use an aggregation pipeline. External callers are unaware.
4. **FKs are adapter-internal protection, not a contract** — business code must not rely on FK errors. Referential integrity is maintained by store methods (e.g. import order: projects → definitions → runs). The portability test: if we disable `PRAGMA foreign_keys`, all business logic should still be correct.
5. **IDs are strings** — `"plan:<ideaId>"`, `"task:<ideaId>:<taskId>"` for definitions; 24-char hex hash for runs. No autoincrement.
6. **Timestamps are ISO 8601 strings** — generated by TS (`new Date().toISOString()`), not by SQLite.
7. **JSON payloads are objects at the port layer** — `WorkflowDefinitionRecord.body` is a string; adapters control serialization.
8. **Transactions are encapsulated** — callers never hold a transaction object; atomic operations are composite port methods (e.g. `importFromJsonl`).

## Phase evolution path

**Phase 1.5a (current):** Bash scripts still write JSONL. SQLite is the read layer, populated by the idempotent importer.

**Phase 1.5b (next PR):** Replace `plan_progress.sh` with a thin TS shim that calls `importFromJsonl` then queries `planProgress()`. SQLite becomes the read source.

**Phase 2 (Wrap):** `create_claude_task.sh` and `dispatch_plan.sh` rewritten as TypeScript modules. Write path moves to ports. JSONL becomes an export, not a source.

## Status: what's NOT built yet

- Quest Store, Object Store, Feedback Store, Audit Store — no Phase 1 producers
- Artifact Store — artifacts live in Git + GitHub (PRs). Adding a table now duplicates truth.
- Session Store — LLM Loop events handled by `src/runtime/events/`
- Phase 2 read-your-writes guarantees (currently import-then-query)

## Follow-up design note (WorkflowRunStatus)

`WorkflowRunStatus` was extended in Phase 1.5 to include Phase 1 dispatch sub-states (`dispatched`, `merged`, `needs_human_review`, etc.) alongside the Phase 0 abstract states (`created`, `running`, etc.). In Phase 2, consider splitting into `lifecycleStatus` (Phase 0 abstract) + `dispatchStatus` (Phase 1 concrete) to avoid coupling the two vocabularies.
